================================================================================
OOXML BUILD PIPELINES - DIRECTORY STRUCTURE ANALYSIS
================================================================================

PURPOSE:
--------
Multi-project workspace for OOXML document generation across various subjects.
Houses projects that programmatically generate PowerPoint presentations and
Excel workbooks using raw OOXML (Office Open XML) manipulation.

CORE ARCHITECTURE:
------------------
Two shared build pipelines used by all projects:
  1. deck_core/     - Raw OOXML PowerPoint (.pptx) generation engine
  2. workbook_core/ - Raw OOXML Excel (.xlsx) generation engine

CURRENT PROJECTS:
-----------------
  вҖў DDG        - US Navy destroyer shipbuilding market analysis
  вҖў Submarine  - US Navy submarine shipbuilding market analysis

SCALABILITY DESIGN:
-------------------
Workspace is architected to support many future projects across diverse domains.
Each new project would follow this standard pattern:
  - {project_name}/deck/          - PowerPoint build (imports deck_core)
  - {project_name}/workbook/      - Excel build (imports workbook_core)
  - {project_name}/research/      - Source data and reference materials
  - Build scripts that orchestrate pipeline with project-specific modules

Example for a new project called "carrier":
  - carrier/deck/build_deck.py           вҶ’ imports deck_carrier.lib
  - carrier/deck/deck_carrier/slides/    вҶ’ Python slide modules
  - carrier/workbook/build_workbook.py   вҶ’ imports workbook_carrier.lib
  - carrier/workbook/workbook_carrier/sheets/ вҶ’ Python sheet modules

KEY CONCEPTS:
-------------
  ACTIVE PIPELINE   = Folders/files imported by build_*.py scripts (MUST NOT MOVE)
  SPECS/DRAFTS      = Text specifications, experimental drafts (can be relocated)
  ARCHIVE           = Historical versions with _archive_* naming
  CORE              = Shared libraries used by all projects
  SUPPORT           = Logs, configs, tooling

STATUS AS OF: June 3, 2026
Recent intensive development (June 1-3) indicates active refactoring phase.
