## Updates I would make

### 1. Fix `sheet_probe.py` so it recognizes total-row input/link styles

This is the most concrete bug I see. `sheet_probe._style_maps()` currently classifies hardcoded inputs as `S_NUM_INPUT`, `S_PCT_INPUT`, and `S_NUM_INPUT_TOTAL`, and green links only as `S_LINK_NUM` and `S_LINK_PCT`.  But `styles.py` defines additional total-row variants: `S_PCT_INPUT_TOTAL`, `S_LINK_NUM_TOTAL`, and `S_LINK_PCT_TOTAL`, and `BORDER_TOP_FOR` maps base input/link styles to those variants for total rows. 

Because `inspect_worksheet_xml()` only rolls a cell into `hardcoded_inputs` or `green_links` when the style index is in those sets, a percent input in a total row or a green cross-sheet link in a total row will be underreported in the probe summary, even though it appears in the raw cells list. 

I would update:

```python
inputs = {getattr(st, n) for n in (
    "S_NUM_INPUT", "S_PCT_INPUT",
    "S_NUM_INPUT_TOTAL", "S_PCT_INPUT_TOTAL",
) if hasattr(st, n)}

links = {getattr(st, n) for n in (
    "S_LINK_NUM", "S_LINK_PCT",
    "S_LINK_NUM_TOTAL", "S_LINK_PCT_TOTAL",
) if hasattr(st, n)}
```

This is a low-risk correctness fix.

### 2. Make worksheet names fail fast instead of silently sanitizing/truncating

`_sheet_name()` currently cleans invalid Excel sheet characters and then truncates to 31 characters before checking duplicates.  That is convenient, but it conflicts with the stated design principle that worksheet names are load-bearing because formulas and accessors reference them by value. The guide also emphasizes that the title banner and tab should not drift and that cross-sheet links should be built through explicit refs/accessors. 

The risk is subtle: if a `TAB_NAME` or `SheetEntry.tab_name` is too long or contains invalid characters, the physical tab can silently differ from the author’s intended formula name. You already reject duplicate names because silent suffixing would break formulas; the same reasoning applies to silent sanitization and silent truncation.

I would change `lib.py` so explicit `TAB_NAME` / `SheetEntry.tab_name` values must already be legal Excel sheet names. For module-derived fallback names, either keep sanitization or also fail fast; I’d prefer fail-fast everywhere unless you are relying on filename-derived sanitization today.

Recommended behavior:

```python
raw = str(_item_raw_name(item)).strip().strip("'")
clean = _clean_sheet_name(raw)

if raw != clean:
    raise ValueError(f"Invalid worksheet name {raw!r}; contains Excel-invalid characters")

if len(clean) > 31:
    raise ValueError(f"Worksheet name {clean!r} exceeds Excel's 31-character limit")
```

Keep the case-insensitive duplicate check.

### 3. Validate table names, table headers, table refs, and defined names

The docs say `ExcelTable.name` must follow safe-name rules, headers must be unique and non-empty, and defined names must follow Excel naming rules.  The guide repeats that defined names should follow Excel’s naming rules and warns about sheet-scoped name traps.  But the packager currently mainly checks for duplicate table names and duplicate defined names while rendering.  `build_defined_names_xml()` documents the naming rules but does not enforce them. 

I would add validators to `lib.py` or `tables.py` and call them inside `package_workbook()`:

```python
validate_defined_name(name)
validate_table_name(t.name)
validate_excel_table(t)
```

Checks worth adding:

* table name is non-empty, starts with a letter or underscore, contains only allowed characters, is not a cell-reference shape, and is workbook-unique case-insensitively;
* table headers are non-empty and unique case-insensitively;
* table `ref` is a valid A1 rectangular range and its column count equals `len(headers)`;
* `t.totals.keys()` are a subset of `headers`;
* defined names follow `[A-Za-z_][A-Za-z0-9_.]*`, are not cell refs, and do not start with reserved `_xlnm`.

This is directly analogous to the deck refactor’s philosophy: if the build surface documents a contract, the packager should enforce it.

### 4. Make `build_table()` reject over-wide rows

`build_table()` validates that `col_styles` length equals `headers`, but if a data row has **more values than headers**, it slices the row to `len(headers)` and silently drops the extra values.  That is exactly the kind of failure mode a raw-OOXML build pipeline should avoid.

I would add:

```python
for i, dr in enumerate(data_rows):
    if len(dr) > len(headers):
        raise ValueError(
            f"data row {i} has {len(dr)} values but table has {len(headers)} headers"
        )
```

before padding/truncation. Silent truncation is more dangerous than a build error.

The workbook side still repeats `XML_DECL` / namespace constants across `styles.py`, `primitives.py`, `sheet_probe.py`, and imports them indirectly through `primitives.py`.

```python
# workbook_core/ooxml.py

```

Then import from it in `styles.py`, `primitives.py`, `tables.py`, `lib.py`, and `sheet_probe.py`. This is a maintainability cleanup, not a behavioral change.

LLM: Yes, the current workbook pipeline **does mandate some workbook structure**, but it **does not mandate specific worksheet tab names** like “Inputs,” “Control Panel,” “TAM Model,” etc.

It **does** currently give some implicit guidance toward space-separated tab names, but that guidance is not explicit enough, and a few examples still use underscore-heavy tab names. I would update the docs/snippets to make your preference unambiguous.

## What the pipeline currently mandates

The pipeline mandates **logical tab groups**, not tab names.

Every sheet item must declare a group via `SHEET_GROUP` or `SheetEntry.group`, and the group must be one of the keys in `groups.SHEET_GROUPS`: `guide`, `inputs`, `data`, `model`, `outputs`, `validation`, or `sources`. The groups also control tab color and canonical block order. The packager checks that all sheets declare a group, that the group is known, that each group appears in one contiguous run, and that those runs follow the canonical group order.  

So the current pipeline **does impose a workbook chapter taxonomy**:

`guide → inputs → data → model → outputs → validation → sources`

But it does **not** say a workbook must have a tab literally named `Inputs`, `Data`, `TAM_Model`, `Sources`, or anything similar. The groups are internal organizational metadata, not sheet names.

## What the pipeline currently does with tab names

For the one-module-per-tab style, `TAB_NAME = None` means the tab name is derived from the Python filename by replacing underscores with spaces and applying title case. For example, `pop_coefficients.py` becomes `Pop Coefficients`.  The guide says the same thing and tells authors to set `TAB_NAME` explicitly only when they want a name the filename cannot express, with examples like `"SCN Annual"` and `"Prime 10-K"`. 

For the consolidated style, `SheetEntry(tab_name=..., group=..., render=...)` requires the caller to provide the tab name directly. `SheetEntry` exists so one source file can register multiple tabs; the `tab_name` is described as load-bearing because formulas and accessors reference it by value. 

The template also mirrors the packager: the title banner derives from `TAB_NAME` or the filename in title case, so the row-2 banner and the physical worksheet tab stay in sync. 

So: **no fixed tab names are mandated**, but **tab names are load-bearing once chosen**.

## Where the current pipeline does suggest tab names

It gives examples, not requirements.

The guide’s examples include filename-derived names like `Pop Coefficients`, explicit names like `SCN Annual` and `Prime 10-K`, and accessor examples like `Control_Panel`, `TAM_Model`, and `POP_Location_Parse`.  The snippets also use examples such as `TAM_Model`, `POP_Location_Parse`, and `POP_Coefficients`. 

Those underscore examples are **not mandatory**, but they are a real authoring nudge. If you want agents/authors to avoid underscores in visible tab names, I would clean those up.

## Does the current pipeline guide formatting of tab names?

Partly, but not explicitly enough.

The strongest existing guidance is the default behavior: module filenames use underscores because Python needs them, but visible Excel tab names convert those underscores to spaces. `budget_normalized.py` would become `Budget Normalized`, not `Budget_Normalized`.  The guide also says the registry has “no name to register; the module owns its name,” which reinforces that visible names are owned at the sheet module level rather than centrally prescribed. 

But I do **not** see an explicit style rule like:

> Use short, human-readable tab names with spaces, not underscores. Reserve underscores for Excel defined names, table names, and Python identifiers.

That is the rule you want, and I would add it.

## Important distinction: tab names vs Excel names

I would keep underscores for **defined names** and **native table names**.

The workbook docs correctly require safe Excel identifier names for defined names and table names, such as `portfolio_tam` or `tbl_subawards`. Those are not visible worksheet tabs; they are formula/table identifiers where spaces are not appropriate.  

So the convention should be:

**Visible worksheet tabs:** `Control Panel`, `TAM Model`, `POP Coefficients`, `Budget Normalized`
**Python modules:** `control_panel.py`, `tam_model.py`, `pop_coefficients.py`
**Defined names / table names:** `portfolio_tam`, `tbl_subawards`, `tam_total`

## On “role” and “tokens”

I do think there is a place for **tokens** in the workbook pipeline. I do **not** think there should be a visible “role” concept in the generated workbook.

The current workbook pipeline already has token-like pieces:

* `S_NUM_INPUT`, `S_NUM`, `S_LINK_NUM`, `S_PCT`, `S_TOTAL`, etc. are really **cell-purpose tokens**, not just visual styles. They encode hardcoded input vs formula result vs cross-sheet link vs total row. 
* `SHEET_GROUP` is already a **sheet-purpose / ordering token**, but it should remain internal metadata, not a visible label. 
* The sheet guide already tells authors to pick styles by cell purpose and explains the finance-model color semantics. 

So my recommendation is:

**Use tokens internally. Avoid “role” language in visible workbook output.**

For example, this is good:

```python
styles=[S_DEFAULT, S_NUM_INPUT, S_DEFAULT]
```

This is not good as visible workbook text:

```text
Role: Input Surface
Role: Model Engine
Role: Deck-Facing Contract
```

Internally, “role” can be called **purpose** or **cell purpose** in the docs. That is more Excel-native and less AI-framework-ish.

## What workbook polish should mean

For workbooks, polish is not the same as slide polish. In a deck, polish often means stronger visual hierarchy, role-based typography, controlled fills, commentary rails, and careful composition. The deck pipeline now has explicit role tokens for typography and insets because slide authors are placing shapes on a canvas. 

In Excel, polish should mean:

* compact labels;
* consistent numeric semantics;
* no over-explaining;
* no artificial guide/readme tabs;
* title banners that mirror tab names;
* section headers that are short and functional;
* inputs visibly blue, links visibly green, formulas black;
* total rows with clean divider rules;
* native Excel tables only where filters/structured references are useful;
* source comments/docstrings can be verbose, emitted cells cannot.

That is already mostly the direction of the workbook style guide: no gridlines, no wraps, no merges, no notes/quality columns except a narrow exception for Sources, explicit styles, total-row discipline, and Arial 8 throughout.

* visible phrases like “This workbook,” “This sheet,” “generated,” “LLM,” “AI-generated,” “below are,” or “use this tab”;
