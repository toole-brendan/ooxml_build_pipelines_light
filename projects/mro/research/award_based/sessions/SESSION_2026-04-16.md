# Session 2026-04-16: Reducing Unclassified Services TAM

## Goal

Starting point: Services TAM = $9.5B, of which ~$7B (~74%) was "Unclassified" by vessel. Objective was to push more $ into specific-hull or vessel-type buckets so the Services sheet reports more useful TAM breakdowns.

## Work Completed

### 1. Per-mod regex sweep (v1 explosion)

The original `classify_vessel()` only consulted the award-level `description`. Per-modification transaction data (FPDS raw mods) was already cached on disk at `data_pull/output/fpds/raw/*_mods.json` but never consulted. Mod-level text is often richer ("USS GREEN BAY (LPD 20) FY25 DSRA" instead of the award-level "Ship repair services").

**Built**: `data_pull/vessel_explode.py`
- Walks all FPDS per-mod records for each services-PSC award
- Applies `VESSEL_DESC_PATTERNS` to each mod description
- Buckets each mod's `obligated_amount` by matched hull
- Emits one exploded row per (award, detected hull) + one residual row for no-match mods
- Dedup key: `(piid, agency_id, mod_number, date_signed)`

**Built**: `sheets/awards_by_hull.py`
- New "Awards By Hull" sheet, parallel to Awards Data
- Excel Table name `AwardsByHull`, services-only rows
- Preserves v1 Awards Data sheet intact

**Wired**: `build_from_data.py` to call `create_awards_by_hull(wb)`.

**Result**: v1 explosion moved ~$940M Navy services out of Unclassified ($6.96B -> $6.02B).

### 2. Regex bug fix

Found a false positive in `VESSEL_DESC_PATTERNS`: `vcs\b` (intended for "Virginia-Class Submarine") matched the common abbreviation "SVCS" (services). "SUBMARINE PRESERVATION MAINTENANCE SVCS" was falsely tagged Virginia-class. Fix: `vcs\b` -> `\bvcs\b` in `data_pull/classify_awards.py`.

### 3. SAM.gov exploration (dead-end)

Tried pulling RFP/SOW synopsis text from SAM.gov for the residual PIIDs.

- Opportunities API: 1,000 req/day cap for registered keys - 1,592 unique residual solicitations would take 2 days and is fragile
- Found the public bulk CSV at `https://falextracts.s3.amazonaws.com/Contract%20Opportunities/datagov/ContractOpportunitiesFullCSV.csv` (~212MB, no API key required, daily refresh)
- Downloaded to `data_pull/output/sam/ContractOpportunitiesFullCSV.csv`
- **Only 14 of 1,592 target solnums matched (0.9% of residual $)** - SAM.gov archives most historical solicitations out of this snapshot

The bulk CSV remains cached for future use on live/active solicitations.

### 4. Parent IDV enrichment

Services work is heavily task orders under IDVs. The IDV itself often has vessel context in its title ("East Coast General Ship Repair IDIQ", "PIA/CIA PLANNING/EXECUTION SAN DIEGO CARRIERS"). FY2025 FPDS pull only captured task orders, not the older IDVs themselves.

**Built**: `data_pull/enrich_idv_descriptions.py`
- Collects all `referenced_idv_piid` values from services-PSC mods
- Queries FPDS Atom Feed per PIID with MOD 0 filter
- Extracts `descriptionOfContractRequirement`, vendor, PSC, DAP, claimant, contracting office
- Caches per-PIID to `data_pull/output/fpds/idv_cache/{piid}.json`
- Writes aggregate `data_pull/output/fpds/idv_descriptions.json`
- Ran against 1,411 unique IDVs (~28 min pull)

### 5. Vessel Type (super-group) classification

Added a coarser classification tier that captures $ where we can tell the vessel family but not the specific hull (e.g., "LCS CLASS SHIPS", "SUBMARINE MAINTENANCE", "WEST COAST CARRIERS", "USNS BIG HORN").

**Modified**: `data_pull/classify_awards.py`
- `VESSEL_TO_SUPERGROUP` dict: maps each vessel_class -> type-level label from Vessel Taxonomy sheet
- `SUPERGROUP_DESC_PATTERNS`: ordered regex list matching broader references
- `classify_vessel_supergroup(description)` helper
- Labels align with the Vessel Taxonomy sheet's Type-level categories: Aircraft Carriers, Surface Combatants, Submarines, Amphibious Warfare Ships, Mine Warfare, Combat Logistics Ships, Expeditionary & Seabasing Ships, Surveillance Ships, Salvage Ships & Fleet Ocean Tugs, Cutters, Unmanned Maritime Platforms, Auxiliary Ships, Combatant Crafts, Support Crafts, Multi-class

**Built**: `data_pull/vessel_explode_v2.py`
- Four-tier classification cascade per mod:
  1. Specific hull via mod description
  2. Specific hull via parent IDV description
  3. Vessel Type (super-group) via mod description
  4. Vessel Type via parent IDV description
  5. Else -> residual
- `_normalize_to_deduped()` post-processing: forces each PIID's exploded row sum to match the deduped canonical `fy2025_obligation` exactly. Handles the edge case where FPDS raw mods occasionally have duplicate records with different obligated amounts.
- Writes to `navy_services_awards_by_hull_v2.json` / `cg_services_awards_by_hull_v2.json` (v1 JSONs untouched).
- Output row schema adds: `vessel_supergroup` (stored under that internal key; displayed as "Vessel Type" in the workbook), `hull_source` (mod_description | idv | supergroup_mod | supergroup_idv | residual | no_mods_found), `parent_idv_piid`, `parent_idv_description`.

**Result**:
- Navy services: 31% -> 39.8% classified (from $2.94B to $3.82B)
- CG services: ~0% -> 37.7% classified (from $103K to $176M) - huge lift because v1 had no CG-specific patterns
- Net lift across both services: about $1.06B newly classified

### 6. Services sheet rewire and layout

Repointed the Services sheet from the `Awards` table to the `AwardsByHull` v2 table. Supports two parallel views that both reconcile to the same grand total:

- **By Vessel Type** (combined Navy + CG, one table) - broader buckets, full rollup
- **By Hull Program** (Navy table + CG table) - specific hulls

**Modified**: `sheets/product_procurement.py`
- Parameterized `_tam_formula()`, `_write_condensed()`, `_write_crosstab()` with `table=`, `category_field=`, `psc_descriptions=` so Services can share the helpers while pointing at AwardsByHull
- `_write_condensed` and `_write_crosstab` single-category mode: each emits one category column per entry + one Unclassified column (residual). No mixed-mode, no double-count risk.
- `get_hull_programs(service, psc_codes, source='awards'|'awards_by_hull')` - sort source matches the data source the table actually queries
- New `get_vessel_types(psc_codes)` - sorted dynamically from AwardsByHull
- New `_load_awards_by_hull()` with v2-preferred / v1-fallback logic

**Modified**: `sheets/services.py`
- Vertical layout: Vessel Type table (combined Navy+CG) FIRST, then Navy hull, then CG hull, then Condensed TAM/SAM sections (same Vessel Type-first order)
- Dynamic column sort: all category columns ordered biggest-to-smallest left-to-right by $ (sourced from the SAME table each sheet queries)
- Column labels: "Vessel Type" (not "Vessel Supergroup"), "Unclassified" (not "Residual") - consistent across all tables

**Modified**: `sheets/awards_by_hull.py`
- AwardsByHull column header for `vessel_supergroup` is "Vessel Type" (data key unchanged, display label only)

## Potential Next Steps on Unclassified Residual

Navy services residual (truly unclassified, no hull or vessel type) is currently ~$5.78B. Ordered by expected lift-per-effort:

### Near-term, structured-data approaches

1. **Expand VESSEL_DESC_PATTERNS with ship proper names.** The current regex catches class namesakes ("USS ZUMWALT" -> DDG-1000) but not named siblings ("USS ABRAHAM LINCOLN" -> CVN). A curated map of ~300-500 active USN/MSC/USCG hulls by proper name would catch most explicit references. One-time research task.

2. **CG-specific regex.** CG hulls showed up in data (WMEC, NSC, WLB, WLM, WLR, WLBB, WAGB, WLIC, WTGB, WYTL, WLI) but the specific-hull classifier has limited patterns for them. Adding `\bwmec\s*\d|famous[\s-]class|reliance[\s-]class`, `\bwmsl\b|legend[\s-]class`, etc. would move $ out of generic "Cutters" supergroup into specific cutter classes.

3. **Recipient + geography priors.** Apply weak classification based on contractor + place-of-performance when description is silent:
   - Electric Boat + CT/RI -> SSN/SSBN
   - HII Ingalls + MS -> LPD/LHA/DDG
   - BIW + ME -> DDG-51
   - NASSCO + CA -> T-AO/T-AKE
   - Austal + AL -> LCS
   - Fincantieri Marinette + WI -> FFG-62
   - Pearl Harbor Naval Shipyard + HI -> SSN/CVN (Pacific Fleet)
   - Kings Bay + GA -> SSBN

   Implement as a fifth classification tier after the four current ones; emit with confidence `low` and `hull_source='prior'`.

4. **DAP / claimant program enrichment on residual PIIDs.** Many residual services awards may have populated `dod_acquisition_program_description` on their parent IDV (already in our `idv_descriptions.json`) but we only run text regex against the description, not the DAP field. Add DAP lookup as a fast path before supergroup regex.

5. **Expand PIID_TO_VESSEL manual map.** Current manually-curated map has ~100 entries. Sort top 200 residual PIIDs by $ and manually research (DSCR awards DB, press announcements). High quality but labor-intensive. Probably closes the long tail of big-dollar residual PIIDs.

### Data-source expansion

6. **Call `usa_client.get_subawards()` for big residual primes.** The function exists in `data_pull/usa_client.py:448-497` but is never invoked. Subaward narratives from HII/BAE/NASSCO/GD often name the hull being fed. Target prime contracts with >$50M total subaward $.

7. **Historical mod context.** Pull FY23/FY24 mods for same PIIDs - a ship being repaired in FY25 was usually also being worked on in prior years, and older mods often carry cleaner vessel references.

8. **SAM.gov historical API pull (year-by-year).** Abandoned in this session due to rate limit friction, but ~150 calls spread over a few days would cover FY22-FY25 Navy solicitations. Alternative: request a system-account API key for higher rate limits.

### LLM-assisted final sweep

9. **Haiku pass on residual tail.** Feed residual rows (description + PSC + recipient + pop_state + parent IDV description) into a constrained-output classifier. Probably $1-5 total cost for the whole residual. Expected to close most of the remaining tail that doesn't fit regex patterns. Output schema: specific hull / vessel type / none + confidence + reasoning citation.

### Order of recommended work

If picking only two next steps: do **#3 (recipient+geography priors)** as a structured tier (cheap, covers a large portion of ambiguous-description long tail) and **#9 (LLM sweep)** as the final layer on whatever remains.

### What not to re-attempt

- SAM.gov bulk CSV for historical solicitations (proven <1% coverage)
- Per-solnum SAM.gov API lookups (too slow under 1,000/day rate limit for the 1,592 unique solnums)
